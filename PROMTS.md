# Промты, которые помогли довести проект до результата

Ниже набор рабочих промтов, которые я ставил AI.

## 1. Сначала разобраться в задании

```text
Изучи задание README.md, составь план реализации правок
```

## 2. Найти причину падающего теста

```text
Запусти тесты и найди настоящую причину падения
Не подгоняй тесты под текущий код
Какая ошибка ривела к неправильному ответу?
```

## 3. Починить генератор минимально

```text
Проверь app/generator.py
Нужно, чтобы answer реально решал statement
Исправь только математическую причину
Запусти тесты генератора
```

## 4. Реализовать независимый verifier

```text
Реализуй app/verifier.py
Нельзя брать answer генератора
Покрой тестами
```

## 5. Разделить роли verifier и LLM

```text
Проверь app/grader.py
Вердикт is_correct должен брать только verifier
LLM нужна только для понятного объяснения, почему ответ неверный
Не учитывай ответы FakeLLM
Добавь тест, что FakeLLM не может сделать неверный ответ верным
```

## 6. Подключить реальный LLM-провайдер

```text
Добавь поддержку LLM gigachat
Документация https://developers.sber.ru/docs/ru/gigachat/individuals-quickstart.md
FakeLLM оставь провайдером по умолчанию, чтобы тесты работали офлайн
Ключи читать только из переменных окружения
Секреты не коммитить
Добавь пример env в .env.example
```

## 7. Проверить соответствие заданию

```text
Проведи ревью приложения на соответствие README с заданием
Ничего не меняй
Верни список несоответствий по приоритету: что критично, что желательно, что уже сделано хорошо
```

## 8. Заполнить итоговые заметки

```text
Заполни NOTES.md как итоговый отчёт по тестовому заданию
Нужно указать:
- как запустить
- что было сделано
- почему падал исходный тест
- почему нельзя было чинить тест вместо кода
- отформатируй блок ошибки AI, приведи к общему стилю
- что проверено
```

## 9. Добавить PostgreSQL-хранилище

```text
Добавь хранилище с двумя режимами: in-memory по умолчанию и PostgreSQL при настройках POSTGRES_
Нужны таблицы для задач, попыток проверки, примеров и событий
Миграции реализуй через Alembic
Базовый запуск без БД, тесты должны остаться офлайн
```

## 10. Документировать работу с БД

```text
Проверь новую реализацию с БД
Дополни NOTES.md: зачем нужна БД, что хранится в таблицах, как работает выбор
memory/postgres, как запустить через docker compose и как запустить только БД +
локальный uvicorn
```

## 11. Добавить историю попыток и кэширование LLM

```text
Добавь историю проверок по задаче и кэширование feedback
Если тот же task_id, тот же ответ ученика и тот же llm_provider уже проверялись - не вызывай LLM повторно
При этом новую попытку всё равно сохрани в историю
Добавь endpoints для просмотра попыток и отдельной попытки
```

## 12. Проверить runtime-сценарий истории и кэша

```text
Проверь работу через HTTP
создай задачу, два раза отправь один и тот же неверный ответ, проверь что ответы
одинаковые, попыток две, у второй duration_ms=0, а в событиях есть grade_cached
После проверки обнови NOTES.md
```

## 13. Улучшить валидацию пользовательского ввода

```text
Улучши валидацию запросов
Нужно отклонять пустой ответ, оличающиеся разделители вместо ';', нечисловые значения,
пустые части вроде, дубли и слишком длинный ответ
Ошибки API должны быть в едином формате
Проверь это тестами и через HTTP
```

## 14. Добавить понятный feedback без LLM для неверного числа корней

```text
Если ответ валидный по формату, но ученик указал неправильное количество корней, не вызывай LLM
Верни короткий шаблонный feedback: сколько корней должно быть и как записать ответ
```

## 15. Добавить timeout/retry/rate limit для LLM

```text
Добавь политику для GigaChat:
- timeout
- max retries
- retry backoff
- клиентский rate limit
Все значения читай из env
FakeLLM не должен их использовать и тесты должны оставаться офлайн
```

## 16. Добавить метрики и структурные логи

```text
Добавь метрики:
- Prometheus endpoint /metrics;
- метрики по HTTP-запросам, проверкам, LLM-вызовам, валидации и генерации
- JSON-логи в stdout
- X-Request-ID
- события качества проверки
Проверь тестами и опиши в NOTES.md
```

## 17. Добавить расширенные примеры по ключу

```text
Добавь новые типы задач (линейные и рациональные уравнения)
Они должны быть доступны только по отдельному ключу
Без ключа базовый сценарий должен работать как раньше
С ключом должны быть видны и доступны новые типы
Для PostgreSQL добавь миграцию с примерами
```

## 18. Финальная проверка перед коммитом

```text
Запусти полный pytest и pre-commit
Если formatter что-то изменит, повтори проверки
```
