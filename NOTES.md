# EGE Math Grader (FastAPI + SymPy + LLM)

Сервис генерирует задачи по квадратным уравнениям и проверяет ответы ученика.
Ключевой принцип: **вердикт о правильности вычисляется независимо (через SymPy)**,
а LLM используется только для человекочитаемого объяснения ошибки.

## Как запустить

### Вариант 1: стандартный `venv`

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

### Вариант 2: через `uv`

```bash
uv venv
source .venv/bin/activate
uv pip install -r requirements.txt
uvicorn app.main:app --reload
```

Сервис поднимается на `http://127.0.0.1:8000`, Swagger: `http://127.0.0.1:8000/docs`.

Тесты:

```bash
.venv/bin/python -m pytest -q
```

Локально: **75 passed**.

## Что было сделано

### 1) Починен падающий тест

Причина падения: в генераторе уравнений коэффициент `b` был рассчитан с неверным
знаком.

Для корней `r1`, `r2`:

- правильно: `(x - r1)(x - r2) = x^2 - (r1 + r2)x + r1*r2`
- значит коэффициент перед `x`: `b = -(r1 + r2)`

До исправления был `b = r1 + r2`, поэтому в условии и в `answer` возникала
несогласованность: задекларированные корни не всегда решали выданное уравнение.

Почему нельзя "подгонять тест": тест валидирует математическую корректность
бизнес-логики. Изменение ожиданий теста скрыло бы дефект, но не исправило бы
ошибку для пользователя.

### 2) Реализован независимый верификатор (`app/verifier.py`)

Сделано:

- разбор уравнения из `statement`;
- преобразование в синтаксис SymPy (`^ -> **`, `5x -> 5*x`);
- символическое решение через `sympy.solve`;
- нормализация/сортировка корней;
- проверка корректности `declared_answer` и ответа ученика.

Верификатор используется в двух местах:

- **при генерации** (`POST /tasks`) — задача сохраняется только если ключ сходится;
- **при проверке** (`POST /tasks/{id}/grade`) — вердикт `is_correct` берется только
  из верификатора.

### 3) Подключен реальный LLM-провайдер (`app/llm.py`)

Сделано:

- оставлен `FakeLLM` как дефолт для офлайн-режима и тестов;
- добавлен `GigaChatLLM` (переключение через `LLM_PROVIDER`);
- реализован `get_llm()` с выбором провайдера по env.

В `grader` LLM используется только для текста объяснения ошибки.
Решение "верно/неверно" LLM не определяет.

### 4) Добавлено хранилище с PostgreSQL

Изначально задачи жили только в памяти процесса. Этого достаточно для базового
тестового задания, но у такого режима есть ограничения: после перезапуска сервиса
все созданные задачи пропадают, нельзя анализировать попытки учеников и неудобно
хранить заранее подготовленные примеры.

Для этого добавлена абстракция `TaskRepository` и две реализации:

- `MemoryRepository` — режим по умолчанию без внешних зависимостей, удобен для
  тестов и локального запуска без Docker;
- `PostgresRepository` — постоянное хранилище на PostgreSQL, включается через
  переменную `DATABASE_URL`.

При старте приложения `app.storage.factory.init_repository()` выбирает реализацию:

- если `DATABASE_URL` не задан — создаётся in-memory репозиторий и в него
  добавляются demo-примеры;
- если `DATABASE_URL` задан — запускаются Alembic-миграции, проверяется соединение
  с PostgreSQL и далее все операции идут через БД.

В PostgreSQL создаются таблицы:

- `tasks` — сгенерированные или созданные из примера задания, включая скрытый
  эталонный ответ;
- `grade_attempts` — история проверок: ответ ученика, verdict, feedback,
  LLM-провайдер и длительность проверки;
- `examples` — заранее подготовленные валидные задания, которые можно отдавать
  через API;
- `app_events` — структурные события приложения: создание задачи, ошибка генерации,
  завершение проверки.

Зачем это нужно:

- задачи не теряются после перезапуска сервиса;
- можно смотреть историю задач, попыток проверки и качество feedback;
- можно хранить проверенные примеры отдельно от случайной генерации;
- появляется база для аналитики, аудита и отладки поведения LLM.

Новые API-сценарии:

- `GET /examples` — список демонстрационных примеров без эталонного ответа;
- `POST /tasks/from-example/{example_id}` — создать обычную задачу из заранее
  подготовленного примера;
- `POST /tasks/{task_id}/grade` — теперь дополнительно сохраняет попытку проверки
  в `grade_attempts`;
- `GET /tasks/{task_id}/attempts` — история всех проверок по конкретной задаче;
- `GET /attempts/{attempt_id}` — просмотр результата одной проверки;
- `GET /events?task_id=...&limit=...` — история структурных событий приложения.

### 5) Добавлено кэширование LLM-feedback

LLM не должна вызываться повторно, если для той же задачи уже проверялся точно
такой же ответ ученика тем же провайдером. Это уменьшает задержку, экономит
стоимость внешнего API и делает повторную проверку детерминированной.

Кэш реализован через таблицу/коллекцию `grade_attempts`:

1. `POST /tasks/{task_id}/grade` сначала ищет прошлую попытку по связке
   `task_id + student_answer + llm_provider`.
2. Если попытка найдена, сервис возвращает сохранённые `is_correct` и `feedback`
   без вызова `grade_answer()` и без запроса к LLM.
3. При этом новая попытка всё равно записывается в историю с `duration_ms=0`, чтобы
   было видно, что ученик повторно отправлял тот же ответ.
4. В `app_events` пишется событие `grade_cached`; при обычной проверке пишется
   `grade_completed`.

Важно: кэш не влияет на математический вердикт. Источником истины остаётся
`verifier`, а кэш хранит уже рассчитанный результат для того же входа.

### 6) Улучшена валидация пользовательского ввода и ошибки API

Ответ ученика теперь проверяется до запуска математической проверки и до обращения
к LLM. Это защищает сервис от мусорного ввода, делает ошибки предсказуемыми для
клиента и не тратит внешний LLM-запрос на заведомо некорректный формат.

Валидация вынесена в `app/validation.py` и применяется в `GradeRequest`:

- ответ обрезается по краям (`" 3 ; 2 "` -> `"3;2"`);
- пустой ответ отклоняется;
- максимальная длина ответа ограничена 64 символами;
- корни должны разделяться `;`, а не запятой;
- пустые элементы между `;` запрещены;
- повторяющиеся корни запрещены;
- каждый корень должен быть числом: целым, десятичным или дробью (`1/2`).

Ошибки API приведены к единому формату:

```json
{
  "code": "invalid_answer_format",
  "message": "Укажите хотя бы один корень",
  "field": "answer"
}
```

Коды ошибок:

- `invalid_answer_format` — некорректный ответ ученика;
- `invalid_id` — невалидный UUID в path/query;
- `missing_field` — отсутствует обязательное поле;
- `invalid_query_param` — некорректный query-параметр, например `limit=0`;
- `task_not_found`, `example_not_found`, `attempt_not_found` — сущность не найдена;
- `generation_failed`, `example_invalid` — внутренние ошибки генерации/данных.

Также добавлен отдельный сценарий feedback без LLM, когда формат ответа валидный,
но указано неверное количество корней. Например, для двойной отправки `2;3` на
уравнение с одним корнем сервис отвечает:

```json
{
  "is_correct": false,
  "feedback": "В уравнении один корень. Укажите только его, например: 2."
}
```

### 7) Добавлены лимиты для внешнего LLM

Для провайдера GigaChat добавлена политика `GigaChatPolicy`, которая
управляет сетевым поведением клиента:

- `timeout` — ограничивает время ожидания ответа;
- `max_retries` — задаёт количество повторных попыток для временных ошибок;
- `retry_backoff_factor` — управляет паузой между повторами;
- `rate_limit_requests` + `rate_limit_period_sec` — задают клиентский rate limit.

Rate limit реализован в `RateLimiter` как скользящее окно: если лимит исчерпан,
следующий поток ждёт, пока из окна выпадет самый старый запрос. Это защищает от
случайного всплеска запросов к внешнему API и снижает риск 429/лимитов провайдера.

По умолчанию `GIGACHAT_RATE_LIMIT_REQUESTS=0`, то есть клиентский rate limit
отключён. Это удобно для локального запуска и тестов; при реальном использовании
можно задать, например:

```env
GIGACHAT_TIMEOUT=60
GIGACHAT_MAX_RETRIES=3
GIGACHAT_RETRY_BACKOFF_FACTOR=0.5
GIGACHAT_RATE_LIMIT_REQUESTS=20
GIGACHAT_RATE_LIMIT_PERIOD_SEC=60
```

`FakeLLM` эти лимиты не использует, поэтому офлайн-тесты остаются быстрыми и
детерминированными.

### 8) Добавлены метрики и структурные логи

Для мониторинга качества проверки добавлена наблюдаемость в двух формах:

- Prometheus-метрики на `GET /metrics`;
- структурные JSON-логи в stdout.

Метрики включены по умолчанию (`METRICS_ENABLED=true`). Если их отключить, endpoint
`/metrics` вернёт `404 metrics_disabled`.

Основные метрики:

- `ege_grader_grades_total` — количество проверок с лейблами `is_correct`,
  `verify_status`, `cached`, `llm_provider`, `feedback_source`;
- `ege_grader_grade_duration_seconds` — длительность проверки ответа;
- `ege_grader_llm_calls_total` и `ege_grader_llm_call_duration_seconds` — количество
  и длительность вызовов LLM;
- `ege_grader_verify_duration_seconds` — длительность символьной проверки SymPy;
- `ege_grader_validation_errors_total` — ошибки валидации входных данных;
- `ege_grader_generation_attempts_total` — попытки генерации задач;
- `ege_grader_http_requests_total` и `ege_grader_http_request_duration_seconds` —
  количество и длительность HTTP-запросов.

Структурные логи пишутся через `app.structured_log` в JSON-формате. Каждый HTTP
запрос получает `X-Request-ID` (из входящего заголовка или новый UUID), а в лог
попадают `method`, шаблон `path`, `status_code` и `duration_ms`.

Доменные события (`task_created`, `grade_completed`, `grade_cached`,
`validation_error`) сохраняются в хранилище и пишутся в stdout. Для проверки ответа
в событиях дополнительно есть поля качества:

- `verify_status` — результат независимой проверки (`correct`, `wrong`,
  `wrong_root_count`);
- `feedback_source` — источник объяснения (`correct`, `llm`,
  `template_wrong_root_count`, `default_fallback`, `cached`);
- `llm_duration_ms` — длительность LLM-вызова, если он был;
- `cached` — использовался ли сохранённый результат.

### 9) Добавлены расширенные примеры по отдельному ключу

Базовая задача тестового задания — квадратные уравнения — остаётся доступной без
ключа и работает как раньше:

- `POST /tasks` по умолчанию генерирует `task_type=quadratic`;
- `GET /examples` без ключа возвращает только квадратные примеры;
- ответ ученику не отдаётся в API.

Дополнительно добавлены расширенные типы:

- `linear` — линейные уравнения;
- `rational` — рациональные уравнения.

Они доступны только при наличии корректного заголовка:

```http
X-Extended-Examples-Key: <EXTENDED_EXAMPLES_KEY>
```

Поведение API:

- `GET /examples` без ключа показывает только базовые квадратные примеры;
- `GET /examples` с ключом дополнительно показывает `linear` и `rational`;
- `POST /tasks?task_type=linear` и `POST /tasks?task_type=rational` без ключа
  возвращают `403 extended_access_required`;
- те же endpoints с корректным ключом генерируют расширенные задания;
- `POST /tasks/from-example/{example_id}` для расширенных примеров без ключа
  возвращает `404 example_not_found`, чтобы не раскрывать наличие закрытого примера.

В in-memory режиме расширенные demo-примеры seed-ятся вместе с базовыми. В
PostgreSQL они добавляются миграцией `002_extended_examples`.

### Как запустить с PostgreSQL

Самый простой вариант — через Docker Compose:

```bash
cp .env.example .env
docker compose up --build
```

Compose поднимает два сервиса:

- `db` — PostgreSQL;
- `app` — FastAPI-приложение на `http://127.0.0.1:8000`.

Для запуска только базы и локального `uvicorn`:

```bash
cp .env.example .env
docker compose up db
export DATABASE_URL="postgresql+psycopg://ege:ege_dev@localhost:5432/ege_grader"
uvicorn app.main:app --reload
```

Миграции Alembic запускаются приложением автоматически при старте, вручную их
вызывать не нужно.

## Где AI ошибся и как это было исправлено

1. На раннем проходе AI фокусировался на LLM-вердикте (JSON от модели) вместо
   независимой математики.
   Исправление: вердикт переведен на `verifier`, LLM оставлен только для feedback.

2. AI недооценивал риск "доверять полю `answer` генератора".
   Исправление: добавлена обязательная проверка сгенерированной задачи через
   `verify_task` до сохранения.

3. AI предлагал локальные правки тестов как быстрый путь к "зелени".
   Исправление: устранена корневая причина в формуле генератора (`b` со знаком
   минус), после чего тесты проходят по сути, а не по подгонке.

## Минимальный ручной e2e-check

Проверен реальный сценарий:

1. `POST /tasks` -> возвращает задачу:
   `x^2 + 4x + 0 = 0`
2. `POST /tasks/{id}/grade` с неверным ответом `-4;1` ->
   `is_correct: false` + осмысленное объяснение ошибки.

Это подтверждает, что интеграция API + verifier + LLM работает в рантайме.

Дополнительно проверен сценарий истории и кэша на поднятом сервисе:

1. `POST /tasks` -> создана задача.
2. Первый `POST /tasks/{id}/grade` с ответом `not-a-root` -> `is_correct: false`,
   обычная проверка заняла `793ms`.
3. Второй `POST /tasks/{id}/grade` с тем же ответом -> тот же ответ API, но
   повторная попытка записана с `duration_ms=0`.
4. `GET /tasks/{id}/attempts` -> возвращает обе попытки.
5. `GET /attempts/{attempt_id}` -> возвращает конкретную попытку.
6. `GET /events?task_id={id}` -> есть события `task_created`, `grade_completed`,
   `grade_cached`.

Проверены сценарии валидации и ошибок API:

1. `POST /tasks/{id}/grade` с пустым ответом -> `422 invalid_answer_format`.
2. `POST /tasks/{id}/grade` с `2,3` -> `422 invalid_answer_format` с подсказкой
   использовать `;`.
3. `POST /tasks/{id}/grade` с `abc` -> `422 invalid_answer_format`.
4. `POST /tasks/not-a-uuid/grade` -> `422 invalid_id`.
5. `POST /tasks/{unknown_uuid}/grade` -> `404 task_not_found`.
6. `GET /events?limit=0` -> `422 invalid_query_param`.
7. `POST /tasks/{id}/grade` с лишним корнем для задания с одним корнем ->
   `is_correct: false` и локальный feedback без вызова LLM.

Проверены расширенные примеры:

1. Локально с `EXTENDED_EXAMPLES_KEY=test-extended-key`:
   `GET /examples` без ключа -> только `quadratic`, с ключом ->
   `quadratic`, `linear`, `rational`.
2. Локально с ключом:
   `POST /tasks?task_type=linear`,
   `POST /tasks?task_type=rational`,
   `POST /tasks/from-example/{linear_id}`,
   `POST /tasks/from-example/{rational_id}` -> `200`.
3. Локально без ключа:
   `POST /tasks?task_type=linear` -> `403 extended_access_required`.
4. Docker + PostgreSQL на `:8001`: базовый сценарий `quadratic` работает,
   `answer` не утекает, grade сохраняет попытку, `/metrics` доступен.
